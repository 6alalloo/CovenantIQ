# CovenantIQ Codebase Redundancy and Staleness Audit

Date: 2026-02-26
Audited by: Codex (repository-wide static and reference scan)

## 1. Scope and Method

Scope reviewed:
- Backend: `src/main`, `pom.xml`, runtime static assets
- Frontend: `frontend/` configs, source, and build artifacts
- Documentation: `docs/`, `README.md`, top-level project docs
- Repository hygiene: tracked vs untracked files, generated artifacts, file reference checks

Method used:
1. Enumerated tracked/untracked files and key manifests.
2. Cross-checked file references/imports/usages (frontend and docs).
3. Verified runtime/config behavior where possible (including Vite config resolution signal).
4. Flagged findings by confidence and cleanup risk.

## 2. Executive Summary

High-confidence cleanup candidates were found:
- Generated files are tracked and causing noisy diffs (`*.tsbuildinfo`, generated Vite config artifacts).
- Duplicate Vite config sources exist; runtime loads `vite.config.js`, which can silently shadow `vite.config.ts`.
- Several documentation files are stale and contradict the current implemented system (JWT/RBAC, attachments, comments, activity, etc.).
- At least one untracked media file appears orphaned (`frontend/image.png`) with zero references.

No critical backend Java dead-code cluster was found in this pass.

## 3. Findings

## A. Redundant or Generated Files (High Confidence)

### A1) Tracked TypeScript build artifacts (`*.tsbuildinfo`)
- Files:
  - `frontend/tsconfig.app.tsbuildinfo`
  - `frontend/tsconfig.node.tsbuildinfo`
- Why this is redundant/outdated:
  - These are generated by TypeScript build mode and change frequently with environment/compiler state.
  - They create noisy commits and merge conflicts without source value.
- Evidence:
  - Build mode enabled in `frontend/package.json:8` (`"build": "tsc -b && vite build"`).
  - Existing `.gitignore` has no ignore rule for `*.tsbuildinfo` (`.gitignore:1-9`).
- Recommendation:
  - Stop tracking both files and add `*.tsbuildinfo` to ignore rules.

### A2) Duplicate Vite config sources (`vite.config.ts` + `vite.config.js`)
- Files:
  - `frontend/vite.config.ts:1-12`
  - `frontend/vite.config.js:1-11`
  - `frontend/vite.config.d.ts:1-2`
- Why this is redundant/outdated:
  - Two source-of-truth candidates exist with equivalent config.
  - Runtime resolution signal shows Vite attempts to load `frontend/vite.config.js` first (`npm run dev -- --debug` output: "failed to load config from ...vite.config.js").
  - This can make edits in `vite.config.ts` appear ignored in dev unless JS is regenerated.
- Supporting config signal:
  - `frontend/tsconfig.node.json:3,9` includes `vite.config.ts` with `composite: true`, which generates JS/DTs artifacts.
- Recommendation:
  - Keep one canonical config only (prefer `vite.config.ts` in this TS repo).
  - Prevent emitting config JS/DTS into source root.

### B2) Legacy SPA forwarding paths in backend controller
- File:
  - `src/main/java/com/covenantiq/controller/SpaForwardController.java:10-16`
- Why this may be outdated:

- Recommendation:
  - Reconcile forward list with actual active routes to avoid stale mappings and refresh-path gaps.

## C. Outdated Documentation Files (High Confidence)

### C1) `docs/BRD.md` and `docs/TDD.md` are stale relative to current implementation
- Evidence:
  - `docs/BRD.md:52,58` says mock auth only / real auth out of scope.
  - `docs/TDD.md:26-30` references old `src/features/*` frontend structure that does not exist now.
  - Current implementation includes JWT-authenticated API (`src/main/java/com/covenantiq/config/SecurityConfig.java:58,65`) and real auth flows.
- Recommendation:
  - Archive as historical (`docs/archive/`) or mark clearly as superseded by `FinalBRD.md` and `FinalTDD.md`.

### C2) `docs/PROGRESS_REPORT.md` is stale and has path/text integrity issues
- Evidence:
  - Broken machine-specific link: `docs/PROGRESS_REPORT.md:6` (`c:/Users/6alal/OneDrive/...`).
  - Still describes mock auth as current: `docs/PROGRESS_REPORT.md:42,116-118`.
  - Contains mojibake text: `docs/PROGRESS_REPORT.md:110`.
- Recommendation:
  - Correct or archive; do not treat as current source of truth.

### C3) `docs/backend_completion_plan.md` contains contradictory stale status
- Evidence:
  - Early section claims major items "Completely Missing" including bulk import, user management, attachments, comments, activity (`docs/backend_completion_plan.md:53-61`).
  - Later sections/checklists show completion entries (including 2026-02-25/26) for overlapping areas (`docs/backend_completion_plan.md:95-105`, `277`, `688`, `691`).
  - Current code has those endpoints/services present (examples: `LoanController.java:119,128,134,140,165`; `AttachmentController.java:31,37,43,53`; `UserController.java:24-25,34,50,55`; `ActivityController.java:27,33`).
- Recommendation:
  - Normalize this file into a single current status or archive and replace with a concise up-to-date completion matrix.

### C4) `docs/PHASE_2A_PROGRESS_REPORT.md` and `docs/PHASE_2A_EXECUTION_CUT.md` are historical snapshots
- Evidence:
  - Both files explicitly describe partial/in-progress state as of 2026-02-24 (`docs/PHASE_2A_PROGRESS_REPORT.md` and `docs/PHASE_2A_EXECUTION_CUT.md` headers/status sections).
- Recommendation:
  - Keep as historical artifacts but move to `docs/archive/` or add an "Archived" banner at top.

### C5) README doc pointers likely point to non-canonical docs
- Evidence:
  - `README.md:8` points to `docs/BRD.md` and `docs/TDD.md` (older baseline docs), while richer/current docs exist (`FinalBRD.md`, `FinalTDD.md`).
- Recommendation:
  - Point README to canonical/current docs only.

### C6) Runbook smoke test is outdated for authenticated API
- Evidence:
  - Runbook calls protected endpoints without auth (`docs/RUNBOOK.md:31-49`).
  - Security config requires authentication for `/api/v1/**` (`SecurityConfig.java:65`) and only `/api/v1/auth/**` is public (`SecurityConfig.java:58`).
- Recommendation:
  - Update runbook smoke sequence to include login/token acquisition.
